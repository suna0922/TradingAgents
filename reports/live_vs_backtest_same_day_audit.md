# 今日分析 vs 历史回测 — 逻辑一致性审计

**日期**: 2026-07-22
**问题**: 我今天跑一次 full 分析来指导今日操作，和把这套分析逻辑放到回测里跑历史，结果是否等价？
**结论**: **不等价，存在 1 个交易日的系统性数据偏差。**

---

## 一、核心差异：数据可用窗口差 1 天

### 场景 A：今日 live 分析（你想做的事）

```
2026-07-22 10:00 AM
  → propagate("000423", trade_date="2026-07-22")
  → 内部 agents 调用 get_stock_data(symbol, end_date="2026-07-22")
  → akshare 查询日线：end_date 是今天 → 实际只返回到 2026-07-21（昨天）
  → RSI/MACD/MA 全部基于截至 7/21 的数据计算
  → PM 看到的 close = 7/21 收盘价
```

### 场景 B：历史回测（你的验证方式）

```
回测 run() 遍历到 2026-07-22
  → L1 触发 → propagate("000423", trade_date="2026-07-22")
  → 内部 agents 调用 get_stock_data(symbol, end_date="2026-07-22")
  → akshare 查询日线：end_date 是历史日期 → 返回到 2026-07-22（含当日）
  → RSI/MACD/MA 全部基于截至 7/22 的数据计算
  → PM 看到的 close = 7/22 收盘价（你 live 时还没发生的事情）
```

**偏差量**: 1 个交易日的 OHLCV 数据。若当天涨/跌 5%，所有技术指标都不同。

---

## 二、这个偏差在哪里产生，不在哪里产生

### 2.1 偏差存在的路径

| 数据 | Live（今天） | 回测（历史 7/22） | 差 |
|------|------------|-----------------|-----|
| `get_stock_data(end_date)` | 实际返回到 7/21 | 返回到 7/22 | **1 天** |
| `_load_ohlcv_akshare(curr_date)` | 返回到 7/21 | 返回到 7/22 | **1 天** |
| `build_instrument_context()` | PE/PB 基于 7/21 close | PE/PB 基于 7/22 close | **1 天** |
| MA(close,200), RSI(14) | 基于 7/21 起算 | 基于 7/22 起算 | **1 天** |
| LLM 看到的市场报告 | "最新收盘 7/21" | "最新收盘 7/22" | **不同文本** |

### 2.2 偏差不存在的路径

| 数据 | 原因 |
|------|------|
| 基本面数据 (`get_fundamentals`) | 用 `curr_date` 过滤财报，7/22 vs 7/22 过滤结果相同（同一份季报） |
| 财报发布日 (D2 修复) | Live 用 `curr_date=today` → 最近财报已发布；回测用 `curr_date=7/22` → 同样 |
| PM prompt 模板 | 同一份 system message，不因时间改变 |
| Graph 管线结构 | Bull/Bear/Research/Risk/PM 节点序列相同 |
| 规则格式 (trigger_sql) | 输出 Schema 相同 |

### 2.3 StockTwits/Reddit 差异（不相关）

- Live: 实时数据（但对 A 股已禁用，3-G 修复）
- 回测: 返回 placeholder（已修复）
- **结论**: 对于 A 股，此差异不存在

---

## 三、L2 执行层的差异（更大）

回测不只是"分析"——回测还执行规则：

```
回测 L2:
  7/22 日: 读取 row["close"], row["rsi"], row["ma200"] 
          → 条件 close < MA(close,200) 用 7/22 的数值判断
          → 触发 BUY → 以 7/22 close 成交
```

Live 你想做的事:
```
  7/22 日: 只有 7/21 的数据
          → 无法判断 7/22 的 close < MA(close,200)（因为 7/22 还没收盘）
          → 需要等收盘后才知道
```

**L2 规则执行完全不可比**。回测默认"收盘后运行、收盘价成交"，live 做不到。

---

## 四、量化偏差

| 偏差来源 | 量级 | 方向 |
|---------|------|------|
| OHLCV 少 1 天 | 所有技术指标偏移 1 天 | 如果当天上涨：live 低估、回测正常 |
| 估值快照差 1 天 | PE = close / EPS，close 差 1 天 | ≤ 当天涨跌幅 |
| LLM 文本差 1 天数据 | 分析结论可能不同 | 不可预测 |
| **L2 规则执行** | **完全不可比** | **回测能执行 live 不能** |

---

## 五、正确做法：让回测等价于 live

### 方案：回测中让 L1 分析也看不到"当天"数据

在 `backtest_hybrid.py` 触发 L1 时，传入 `trade_date - 1` 而不是 `trade_date`：

```python
# 当前（不对齐）
result = self.l1_analyzer.run_full_analysis(self.symbol, date_str)
# date_str = "2026-07-22" → graph 内部拿到 7/22 的数据

# 应改为（对齐 live）
analysis_date = self._get_prev_trading_day(date_str)
result = self.l1_analyzer.run_full_analysis(self.symbol, analysis_date)
# analysis_date = "2026-07-21" → graph 内部只能拿到 7/21 的数据
```

**为什么这会对齐**: 在 live 中，7/22 早晨你跑分析，内部 `curr_date=today`，数据源只能返回到昨天。回测中显式设 `trade_date=yesterday`，模拟同样的数据可用性。

### 但仍有一个不可消除的差异：L2 执行

即使对齐了 L1 分析的数据窗口，回测中的 L2 规则执行仍然需要"当天"数据来判断规则条件。Live 中你在 10:00 AM 运行分析得到规则后，L2 需要用**盘中的数据**来实时评估——这与回测的"日末一次性判断"本质不同。

**这部分差异属于日频回测 vs 实时交易的固有限制**，已在 `reports/next_day_open_execution_spec.md` 中详细说明。

---

## 六、对齐后剩余差异清单

| 差异 | 可修复？ | 方案 |
|------|---------|------|
| L1 分析数据窗口差 1 天 | ✅ | 回测 L1 用 `date_str - 1` |
| L2 规则执行时机 | ❌ 日频的限制 | 次日 open 成交方案可缓解 |
| LLM 采样随机性 | ❌ 不可消除 | 设置 seed/temperature=0 可减少 |
| 新闻/公告数据 | ⚠️ 部分 | Yahoo Finance 有历史新闻 |
| Memory/Reflection | ✅ | 1-H 修复已对齐 |

---

## 七、结论

| 问题 | 答案 |
|------|------|
| 今日分析和回测历史结果等价吗？ | **不等价，差 1 天数据窗口** |
| 能修复吗？ | **L1 分析可以**（回测中用前一天日期触发 L1）。**L2 执行无法完全对齐**（日频回测的固有限制） |
| 修复后回测还能验证策略吗？ | **可以，且更可信**——因为回测决策基于和 live 相同的数据窗口 |
| 当前回测结果是高估还是低估？ | **取决于行情方向**。如果标的趋势向上，回测多看到一日上涨 → 高估收益；趋势向下 → 低估 |

*注：本审计聚焦"同一天运行分析 → 同一天指导操作"语义。若你的实际场景是"今天分析 → 明天开始执行"，则差异本身就是你想要的行为（今天分析包含今天的数据作为决策依据）。*
