## 二、Web 端仍未修复的 5 项差距

### WEB-1: 缺少 Sentiment + News 分析师

```python
# graph_adapter.py:298 — 仍只选了 2 个
selected_analysts=["market", "fundamentals"],
```

应改为 `["market", "social", "news", "fundamentals"]` 与主程序默认一致。当前缺失情绪面和新闻面两个信息维度。

### WEB-2: LLM 模型未分化

```python
# graph_adapter.py:293-294 — 两个模型完全相同
config["deep_think_llm"] = "deepseek-chat"
config["quick_think_llm"] = "deepseek-chat"
```

主程序会给基本面分析师和研究经理分配更强的推理模型。Web 所有角色吃同一碗饭，核心分析质量打折扣。

### WEB-3: 大师配置未注入

前端有大师拖拽选择 UI，但 `graph_adapter.py` 创建 `TradingAgentsGraph` 时用 `DEFAULT_CONFIG`，未传入用户选择。选大师的 UI 等同于摆设。

### WEB-4: 旧 `RoundtableSession.run()` 未删除

`agent_service.py:233-402` 保留了 ~170 行旧的独立分析管线。虽然 SSE 路由已切到 `graph_adapter`，但保留死代码增加了：
- 维护负担（两套分析逻辑并存）
- 被意外调用的风险

### WEB-5: 前端数据重复获取

前端仍通过 `GET /api/stock/{ticker}/technicals` 和 `GET /api/stock/{ticker}/fundamentals` 独立获取数据，同时 `graph_adapter.py` 的 `_fetch_data()` 也获取了相同数据（且走的是同源函数）。数据被获取了两次。

## 六、二审新增发现

### NEW-1: `_calc_sma` 在 Web 面板有类型歧义

```python
# graph_adapter.py:383-391
def _calc_sma(series, window: int) -> float:
    s = series["Close"] if isinstance(series, pd.DataFrame) else series
    return round(float(s.tail(window).mean()), 2)
```

作为 `_calc_sma(ohlcv["Close"], 12)` 调用时传入的是 Series，逻辑正确。但作为 `_calc_sma(ohlcv, 5)` 调用时传入 DataFrame，逻辑也正确。**目前所有调用点都匹配了正确的参数类型**，但函数签名不够明确（`series` 可以是 DataFrame 也可以是 Series），后续维护者容易用错。建议拆分为两个函数或使用类型注解。

### NEW-2: 新增代码中 `_SIGNAL_MAP` 未被使用

```python
# graph_adapter.py:34-44
_SIGNAL_MAP: Dict[str, str] = { ... }  # 定义了但从未引用
```

实际使用的是 `_extract_signal_from_state()`（第 47 行）。`_SIGNAL_MAP` 是死代码。

### NEW-3: 服务器日志中也暴露 traceback

```python
# analyze.py:61-62
logging.getLogger(__name__).error(
    "Session creation failed for ticker=%s: %s\n%s", ticker, e, traceback.format_exc()
)
```

如果服务器日志可被非授权人员访问，这会暴露文件路径和内部结构。

## 八、总结

一审报告指出 41 个问题，二审验证 49 个（新增 8 个为细节子项拆分+新增发现）。

**1 个核心架构问题已修复**（Web 与主程序逻辑统一），这确实是最大的进步。但：

- 回测系统 13 个 Bug 纹丝不动，其中 2 个是 CRITICAL（止损裸奔、规则静默失败）
- 调优系统 3 个 CRITICAL Bug 全部健在，OPT-1（返回 None）意味着**你的调优系统从来没有产出一个有效的调优结果**
- Web 安全面 13 个漏洞一个没修，Web 在公网环境下等同于裸奔
- `graph_adapter.py` 虽然漂亮地解决了核心架构问题，但仍有 3 个小差距（分析师不全、模型未分化、大师不注入），修复成本都很低

**一句话：架构对了，但子弹还在枪里没打出去。**
