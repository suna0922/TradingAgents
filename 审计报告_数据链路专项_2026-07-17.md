# 数据链路专项审计报告 — 为什么 Web 面板数据出不来

> 审计日期：2026-07-17 | 焦点：用户输入股票代码后，基本面/技术面数据无法快速展现的根因

---

## 一、直接回答：你的代码到底怎么了

你的期望流程是：

```
输入代码 → 面板立即显示基本面/技术面 → 再启动 LLM 分析
```

实际代码的流程是：

```
输入代码
  → 创建会话（get_stock_name 被阉割成直接返回代码，股票名永远显示不出来）
  → 前端并行请求 /technicals 和 /fundamentals
       → 每个请求 spawn 一个全新 Python 子进程（冷启动 import akshare 就要 5-15 秒）
       → 两个子进程做的是【完全相同】的全量取数工作（100% 重复劳动）
       → 每个子进程内部串行调 10+ 次上游 API，其中最慢的
         stock_financial_analysis_indicator（新浪逐年爬取）被重复调用 4 次
       → 单进程实际耗时 35~100 秒，但超时设的是 60 秒
       → 超时 → 异常被 except 静默吞掉 → 返回全 0 数据 → 面板"看起来坏了"
  → 同时前端打开 SSE 分析流
       → SSE 循环里 q.get(timeout=0.3) 是同步阻塞调用，把整个 uvicorn
         事件循环卡死几分钟
       → 即使子进程算完了，/technicals 的 HTTP 响应也发不出去
       → 面板数据被 LLM 分析"堵在门外"
```

**一句话：数据被重复取了 4 轮、最慢的接口被打了 8-12 次、面板响应又被 SSE 阻塞的事件循环堵住 —— 三个问题叠加，面板永远快不起来。**

---

## 二、Bug 清单（按致命程度）

### 🔴 致命级 — 直接导致"数据不显示"

#### #1 SSE 图执行循环阻塞整个 FastAPI 事件循环（第一根因）

`graph_adapter.py:350-356`

```python
while True:
    try:
        item = q.get(timeout=0.3)   # ← 同步阻塞！在 async 函数里！
    except queue.Empty:
        ...
```

`queue.Queue.get()` 是同步阻塞调用。LLM 分析持续数分钟期间，事件循环几乎 100% 被占死 —— **包括尚未返回的 `/technicals`、`/fundamentals` HTTP 响应都发不出去**。这就是"面板转圈圈直到分析结束"的直接原因。

**修复**：
```python
item = await asyncio.get_event_loop().run_in_executor(None, lambda: q.get(timeout=0.3))
# 或者干脆改用 asyncio.Queue + call_soon_threadsafe
```

#### #2 面板端点每次请求 spawn 全新子进程，且两个端点做完全重复的工作

`stock_service.py:136-152`：`_run_fetcher` 每次 `subprocess.run` 一个新 Python 进程。

- 子进程冷启动 + import akshare/pandas/baostock：**5-15 秒**
- `/technicals`（`stock_service.py:45`）和 `/fundamentals`（`:72`）**各 spawn 一个子进程**
- 两个子进程执行的 `data_fetcher.py:60-95 fetch_all()` 完全一样 → 100% 重复
- 无任何跨请求结果缓存（`stock.py:48` 还显式 `Cache-Control: no-cache`）

#### #3 子进程 60s 超时 + 静默吞异常 → 前端显示全 0

`stock_service.py:143`：`timeout=60`，但单子进程实际耗时 35-100 秒。

超时后 `stock_service.py:44-47 / :71-74`：
```python
except Exception:
    return tech   # ← 返回 latest_price=0 的空壳对象，没有任何错误提示
```

用户看到的"全是 0 的坏面板"就是这里来的。**失败被伪装成了成功。**

#### #4 最慢的接口保护最少、调用最多

`akshare_data.py:618, :653`：`stock_financial_analysis_indicator`（新浪财经逐年爬取，最慢的上游）以**完全相同的参数被直接调用两次**，且绕过了 `_safe_call`（无 120s 缓存、无 3s 节流、无重试）。`:674` 的 `stock_fhps_detail_ths` 同样直连。

#### #5 `get_valuation_metrics` 内部再整跑一遍 `get_fundamentals`

`akshare_data.py:849` + `:805`：末尾无条件重跑 `_append_valuation_metrics` → #4 的慢接口再来一轮。叠加 `data_fetcher.py:74 + :90` 的调用序列，**每个子进程触发 4 次 `stock_financial_analysis_indicator`，一次点击总计 8-12 次**。

---

### 🟠 严重级 — 架构性缺陷

#### #6 `get_fundamentals_structured()` 读模块级全局变量 — 并发竞态

`akshare_data.py:558, :692-714, :802`

```python
_last_fundamentals_structured = {}          # 模块级全局

def get_fundamentals(ticker, date):
    ...
    _last_fundamentals_structured = data     # 副作用写入

def get_fundamentals_structured():           # ← 不接受任何参数！
    return _last_fundamentals_structured     # 返回"最近一次"的值
```

在 uvicorn 单进程内，如果会话 A 调完 `get_fundamentals("600519")` 后、读取全局前，会话 B 调了 `get_fundamentals("000858")` —— **A 的面板会展示 B 股票的财务数据**。受影响的调用点：
- `graph_adapter.py:244-245`（Web 面板数据）
- `numeric_validation.py:35-36`（LLM 数值校验会拿错标的数据！）
- `data_session.py:97-98`、`agent_utils.py:93-94`

**修复**：改为 `get_fundamentals_structured(ticker)` 按 ticker 键控的字典。

#### #7 预取链路是死代码 — 设计意图根本没接上

`analyze.py:40-54` 定义了 `_prefetch_and_warm`（会话创建后立即预热数据），但**全仓库没有任何调用点**。注释还宣称"面板 REST 端点通过 DataSession.get_or_create() 获取同一实例"—— 这是过时的谎言，`stock_service.py` 完全不用 DataSession，走的是子进程。

结果：**三条数据链路（面板 REST 子进程、DataSession、graph_adapter fallback）互不共享任何数据**，同一份数据一次点击最多被完整拉取 4 轮。

#### #8 两个并发子进程竞争写同一 CSV 缓存

`akshare_data.py:431, :461`：缓存文件名对同标的当天相同，`to_csv` 无锁。子进程 A/B 同时 miss → 同时 fetch → 同时写 → 缓存文件可能损坏。同时 `_ak_throttle` 的 3 秒节流是**进程内**变量（`:24-26`），跨进程完全失效 → 两个子进程并发轰击东方财富 → 触发反爬 → 进入指数退避重试（`:150-157`）→ 更慢。

### 🟡 中等级 — 功能/正确性缺陷

| # | 问题 | 位置 | 影响 |
|---|------|------|------|
| #9 | `get_stock_name` 被阉割成 `return ticker` | `stock_service.py:27-29` | 股票名永远显示为代码 |
| #10 | `/fundamentals/history` 是空壳，恒返回空数据 | `stock_service.py:125-127` | 前端历史图表永远空白 |
| #11 | MACD 用 SMA 差冒充（应为 EMA 体系），BOLL/KDJ 硬编码 0 | `graph_adapter.py:261-262`, `data_session.py:188-189` | SSE 推送的技术面数据大半是假值 |
| #12 | `_run_fetcher` 每次泄漏一个 ThreadPoolExecutor | `stock_service.py:150-152` | 长期运行资源泄漏 |
| #13 | `_AK_CACHE_TTL=120s` 短于一次 LLM 分析时长 | `akshare_data.py:40` | 分析中期缓存过期重新取数 |
| #14 | `_valuation_cache` 无 TTL 无上限 | `akshare_data.py:818` | 内存泄漏 |
| #15 | `thread.join(timeout=5)` 再阻塞事件循环 5 秒 | `graph_adapter.py:405` | 分析结束时又冻结一下 |

### ✅ 排除嫌疑：前端没问题

前端时序是正确的：await 会话创建 → 不 await、并行发面板请求 → 开 SSE，且有 SSE 兜底合并防覆盖。**慢的根源 100% 在后端。**

## 三、数据被重复获取的完整账单

一次"输入代码点分析"，同一份数据被拉取的轮数：

| 轮次 | 谁在拉 | 路径 |
|------|--------|------|
| 第 1 轮 | `/technicals` 子进程 A | `data_fetcher.py` 全量 fetch_all |
| 第 2 轮 | `/fundamentals` 子进程 B | 同上，与 A 完全重复 |
| 第 3 轮 | SSE 里的 DataSession.prefetch_panel | uvicorn 进程内，与子进程不共享内存 |
| 第 4 轮 | graph_adapter 阶段 0 fallback | 又一遍 get_fundamentals + OHLCV |

其中最慢的 `stock_financial_analysis_indicator` 总调用次数：**8-12 次**。

---

## 四、修复路线图（按顺序做，做完前 3 步面板就能秒出）

### 第 1 步：解除事件循环阻塞（1 行改动，立竿见影）

`graph_adapter.py:352` 改为：
```python
item = await asyncio.get_event_loop().run_in_executor(
    None, lambda: q.get(timeout=0.3)
)
```
做完这一步，面板响应至少不会被 LLM 分析堵住。

### 第 2 步：砍掉双子进程，面板数据一次取、进程内缓存

- `/technicals` 和 `/fundamentals` 共享同一次取数结果（按 ticker 缓存，TTL 建议 10 分钟）
- 用 `run_in_executor` 在 uvicorn 进程内取数（DataSession 已经这么做了，把面板路由接上它）
- 删掉 `data_fetcher.py` 子进程方案，或者保留但只 spawn 一次

### 第 3 步：给慢接口套保护 + 去重

- `_append_valuation_metrics` 里的 `stock_financial_analysis_indicator` 两次同参调用合并为一次，结果复用
- 全部套上 `_safe_call`（缓存 + 节流 + 重试）
- `get_valuation_metrics` 不再整跑 `get_fundamentals`，直接复用 structured 缓存

### 第 4 步：修全局态竞态

`get_fundamentals_structured()` → `get_fundamentals_structured(ticker)`，内部改为 `dict[ticker] = data`。**这个必须修，否则并发时 LLM 会拿着 A 股票的名字分析 B 股票的财务数据。**

### 第 5 步：接上预取链路

会话创建（POST /session）成功后就后台启动取数预热，SSE 打开时数据已就绪。`_prefetch_and_warm` 的骨架已经写好了，只是没人调用它。

### 第 6 步：清理正确性问题

- 恢复真实的 `get_stock_name`（用进程内缓存 + 落盘缓存）
- MACD 改用 EMA 体系，补齐 BOLL/KDJ 真实计算（stockstats 库项目里已有）
- 超时/异常不再静默吞掉，返回明确错误让前端展示"数据获取失败，请重试"

---

## 五、修完后的目标流程

```
输入代码
  → POST /session（毫秒级返回，同时后台预热取数）
  → GET /technicals, /fundamentals（命中预热缓存 → 亚秒级返回）
  → 面板立即渲染 ✅
  → 打开 SSE → LLM 分析开始（graph_adapter 复用已缓存的数据，不再重取）
  → 事件循环不被阻塞，分析期间面板/其他请求正常响应 ✅
```
